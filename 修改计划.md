现有的旅行助手更多是依赖高德 API 提供“硬数据”（坐标、天气）。要增加技术壁垒，你需要引入“软数据”和“状态持久化”。

1. 引入 RAG (检索增强生成) 解决“攻略盲区”

痛点：高德 API 只能查到景点在哪，但不知道“去环球影城怎么避开排队”或“某家苍蝇馆子好不好吃”。

技术栈：LangChain Document Loaders + BGE/OpenAI Embedding + Milvus/Qdrant (向量数据库)。

实现：写个爬虫抓取几百篇马蜂窝/小红书的优质游记，切片后存入向量数据库。在 LangGraph 中新增一个 Local_Expert_Node（本地向导节点）。当用户问“有什么小众打卡点”，Agent 会先去向量库做语义检索（Vector Search），结合 RAG 给出极具人情味的建议，再结合 MCP 查天气和路线。

2. 引入 Long-term Memory (长期记忆) 解决“冷启动”

痛点：现在的系统每次都是健忘的。

技术栈：LangGraph Checkpointer (基于 PostgreSQL 或 Redis) + Profile Schema。

实现：为每个用户分配一个 user_id。系统在后台维护一个用户画像（Profile）。如果用户在对话中提过“我不吃香菜”或者“我带了小孩”，Profile_Update_Node 会悄悄把这些信息更新到数据库。下次规划行程时，Agent 会自动从数据库读取偏好，排雷不合适的餐厅和高强度徒步路线。

3. 演进为真正的多智能体 (Multi-Agent) 协同

实现：打破现在一个 Planner 包揽一切的局面。引入：

Router Agent：意图识别。

Travel Planner Agent：负责调 MCP 查路线和排期。

Budget Controller Agent：专门负责算账，如果 Planner 排的酒店超出了用户的预算，Budget Agent 会打回重做（在图结构中形成一个循环边 Conditional Edge）。
