# AI 智能旅行助手架构说明

## 1. 项目概览

**项目名称：** AI 智能旅行规划助手

**项目定位：** 面向用户个性化旅行需求的 AI Agent 应用。用户输入目的地、日期、交通方式、住宿偏好、预算和兴趣标签后，系统自动生成结构化旅行计划，并支持地图可视化、多轮自然语言精修、用户反馈记忆和图片/PDF 导出。

**核心能力：**

- 基于 LangGraph `StateGraph` 编排多节点 Agent 工作流。
- 使用 `MemorySaver + interrupt + Command` 实现 Human-in-the-loop 多轮精修。
- 基于 LangChain + Qdrant 构建旅行攻略 RAG 语义记忆。
- 使用 SQLite + Qdrant 双写用户反馈，形成用户情节记忆。
- 接入高德地图 POI/天气 REST API，补充真实景点、餐饮、酒店和天气信息。
- 基于 Vue3 + TypeScript 实现行程展示、地图路线、精修输入和导出能力。

**技术栈：**

| 层级 | 技术 |
|------|------|
| 后端框架 | FastAPI, Pydantic, Uvicorn |
| Agent 编排 | LangGraph `StateGraph`, `MemorySaver`, `interrupt`, `Command` |
| RAG/记忆 | LangChain, Qdrant, SQLite |
| LLM | OpenAI-Compatible Chat Completions API |
| 外部数据 | 高德地图 POI/天气 REST API, Unsplash API |
| 前端 | Vue3, TypeScript, Vite, Ant Design Vue |
| 地图与导出 | 高德地图 JSAPI, html2canvas, jsPDF |

## 4. LangGraph 多 Agent 工作流

核心文件：`backend/app/agents/test.py`

核心类：`GraphTripPlanner`

### 4.1 状态定义

`TripGraphState` 是 LangGraph 的状态结构，主要字段包括：

| 字段 | 说明 |
|------|------|
| `request` | 用户旅行请求 `TripRequest` |
| `user_id` | 用户 ID，用于召回情节记忆 |
| `chat_history` | 多轮对话历史 |
| `spending_tier` | 当前消费档次 |
| `rag_knowledge` | Qdrant 检索到的攻略知识 |
| `user_memory` | 用户历史反馈召回结果 |
| `selected_pois` | 已选景点、餐厅、酒店名称 |
| `selected_poi_details` | 高德返回的真实 POI 详情 |
| `mcp_data` | 当前代码中用于承载高德 POI/天气数据的状态字段 |
| `final_plan` | 最终结构化旅行计划 |
| `critic_scores` | 质量审核评分 |
| `critic_feedback` | 质量审核文字反馈 |
| `revision_count` | 自动修订次数 |
| `resume_from` | 用户精修后要恢复到的节点 |

### 4.2 图结构

当前图由 `StateGraph(TripGraphState)` 构建：

```text
START
  -> knowledge_retrieval
  -> poi_selector
  -> gather_info
  -> planner
  -> image_enricher
  -> qa_auditor
      -> [should_revise=True 且 revision_count<2] poi_selector
      -> [审核通过] await_refinement
  -> interrupt(final_plan)

用户精修后：
Command(resume=用户消息)
  -> await_refinement
  -> refine_agent
      -> knowledge_retrieval
      -> poi_selector
      -> planner
      -> await_refinement
      -> END
```

### 4.3 检查点与中断恢复

图编译时使用：

```python
workflow.compile(checkpointer=MemorySaver())
```

每次执行时通过 `session_id` 作为 LangGraph `thread_id`：

```python
config = {"configurable": {"thread_id": session_id}}
```

首次生成时，图执行到 `await_refinement`：

```python
user_message = interrupt(state.get("final_plan", {}))
```

此时 LangGraph 暂停，把 `final_plan` 返回给前端。用户在结果页输入修改意见后，后端通过：

```python
Command(resume=user_message)
```

恢复同一 `thread_id` 对应的图状态，从 `interrupt()` 位置继续执行。

## 5. 节点职责

### 5.1 `knowledge_retrieval`

**作用：** 检索旅行攻略和用户历史经验。

执行逻辑：

1. 根据城市、偏好、补充需求拼接查询文本。
2. 调用 `EpisodicMemoryManager.recall_lessons()` 按 `user_id` 召回历史反馈。
3. 调用 `QdrantSemanticMemory.search_knowledge_async()` 从 Qdrant 检索攻略知识。
4. 将结果写入 `rag_knowledge` 和 `user_memory`。

### 5.2 `poi_selector`

**作用：** 根据用户需求和 RAG 上下文选择候选 POI。

执行逻辑：

1. 使用 LLM 将用户需求转换为地图搜索关键词。
2. 调用高德 POI 接口批量查询景点、餐厅、酒店候选。
3. 对候选进行去重、限量和兜底。
4. 使用 LLM 从候选中筛选最终 POI。
5. 输出 `selected_pois`、`selected_poi_details`、`poi_search_terms`。

### 5.3 `gather_info`

**作用：** 补齐真实 POI 信息和天气信息。

执行逻辑：

1. 优先使用 `selected_poi_details`。
2. 如果没有候选详情，则根据 POI 名称再次调用高德 POI 接口。
3. 调用高德天气接口获取未来天气。
4. 如果 API Key 缺失或查询失败，则使用本地兜底 POI/天气数据。
5. 输出 `mcp_data`，其中包含 `pois`、`weather`、`warnings`。

说明：字段名 `mcp_data` 沿用早期命名，当前代码实际承载的是高德 REST API 数据。

### 5.4 `planner`

**作用：** 生成结构化旅行计划。

执行逻辑：

1. 从 `mcp_data` 中提取景点、餐厅、酒店、天气。
2. 将景点按经纬度距离进行就近排序。
3. 按旅行天数拆分每日景点。
4. 为每天匹配靠近当日景点的午餐/晚餐。
5. 汇总门票、酒店、餐饮、交通预算。
6. 调用 LLM 优化每日描述、整体建议和独家提示。
7. 输出 `final_plan`。

### 5.5 `image_enricher`

**作用：** 图片增强预留节点。

当前节点保持 `final_plan` 透传，前端和 `/api/poi/photo` 可通过 Unsplash 服务按景点名异步获取图片。

### 5.6 `qa_auditor`

**作用：** 反思与质量审核。

执行逻辑：

1. 调用规则审核 `_build_rule_based_critic_scores()`，检查：
   - 单日景点数量是否过多。
   - 同日景点间距离是否过远。
   - 预算是否超过用户预算。
   - 行程是否体现用户偏好。
2. 调用 `_audit_plan_with_llm()` 让 LLM 复核评分。
3. 输出：
   - `geo_score`
   - `budget_score`
   - `preference_score`
   - `critique`
   - `should_revise`
   - `revision_focus`
4. 如果 `should_revise=True` 且 `revision_count < 2`，通过条件边回到 `poi_selector` 自动重跑。
5. 如果通过审核，进入 `await_refinement` 返回给用户。

### 5.7 `await_refinement`

**作用：** Human-in-the-loop 暂停点。

首次到达时：

```text
interrupt(final_plan)
```

图暂停，当前行程返回前端。

用户精修时：

```text
Command(resume=用户消息)
```

图从暂停点恢复，将用户消息写入 `chat_history`，然后进入 `refine_agent`。

### 5.8 `refine_agent`

**作用：** LLM Router 精修意图识别。

执行逻辑：

1. 读取 `chat_history` 中用户最新修改意见。
2. 调用 LLM 输出标准意图 JSON。
3. 如果 LLM 不可用，使用关键词兜底分类。
4. 根据意图设置 `resume_from`：

| 意图 | 含义 | 路由 |
|------|------|------|
| `end` | 用户满意，结束流程 | `END` |
| `adjust_tier` | 调整消费档次 | `poi_selector` |
| `change_poi` | 增删替换景点/餐厅/酒店 | `poi_selector` |
| `modify_schedule` | 调整某天节奏、顺序、轻松程度 | `planner` |
| `regenerate` | 完全重新规划 | `knowledge_retrieval` |

## 6. RAG 与记忆系统

### 6.1 语义知识库 `QdrantSemanticMemory`

文件：`backend/app/memory/qdrant_manager.py`

使用 LangChain 生态组件：

- `TextLoader`
- `RecursiveCharacterTextSplitter`
- `OpenAIEmbeddings`
- `HuggingFaceEmbeddings`
- `QdrantVectorStore`

能力：

| 方法 | 说明 |
|------|------|
| `ingest_local_docs(file_path)` | 加载本地攻略文档，切分为 chunk 并写入 Qdrant |
| `search_knowledge(query, k)` | 同步语义检索 |
| `search_knowledge_async(query, k)` | 异步语义检索，供 Agent 节点使用 |

嵌入模型支持：

- 本地 HuggingFace 模型。
- OpenAI-Compatible Embedding API。
- `EMBED_PROVIDER=auto/local/openai` 控制策略。

### 6.2 用户情节记忆 `EpisodicMemoryManager`

文件：`backend/app/memory/episodic_manager.py`

采用双写结构：

| 存储 | 内容 |
|------|------|
| SQLite | `trip_id`、`user_id`、`city`、`rating`、`feedback_text`、`created_at` |
| Qdrant | 用户反馈文本向量和 metadata |

核心能力：

- `save_experience()`：保存用户评分和反馈。
- `recall_lessons()`：按 `user_id` 过滤，在 Qdrant 中检索与当前请求相似的历史经验。

## 9. 完整链路

### 9.1 首次生成

```text
用户填写 Home.vue 表单
  -> POST /api/trip/plan
  -> FastAPI 创建 session_id
  -> GraphTripPlanner.plan_trip()
  -> LangGraph thread_id=session_id
  -> knowledge_retrieval
  -> poi_selector
  -> gather_info
  -> planner
  -> image_enricher
  -> qa_auditor
  -> await_refinement
  -> interrupt(final_plan)
  -> 返回前端
  -> Result.vue 展示行程和地图
```

### 9.2 多轮精修

```text
用户在 Result.vue 输入修改意见
  -> POST /api/trip/refine
  -> GraphTripPlanner.refine_trip()
  -> Command(resume=用户消息)
  -> LangGraph 从 await_refinement 恢复
  -> refine_agent 识别意图
  -> 根据 resume_from 跳转：
       adjust_tier/change_poi -> poi_selector
       modify_schedule -> planner
       regenerate -> knowledge_retrieval
       end -> END
  -> 重新执行后续节点
  -> qa_auditor
  -> await_refinement
  -> interrupt(updated_plan)
  -> 返回更新后的行程
```

### 9.3 用户反馈记忆

```text
用户提交评分和反馈
  -> POST /api/trip/feedback
  -> EpisodicMemoryManager.save_experience()
  -> 写入 SQLite trip_history
  -> 写入 Qdrant user_feedback_v2
  -> 后续 knowledge_retrieval 按 user_id 召回相关经验
```

## 10. 错误处理与降级

| 场景 | 降级策略 |
|------|------|
| LLM API Key 缺失 | `_request_llm_json()` 返回空，使用规则兜底 |
| LLM 返回非 JSON | `_extract_json_object()` 尝试抽取 JSON 对象 |
| 高德 API Key 缺失 | 使用本地兜底 POI 和天气 |
| 高德 POI 查询失败 | 写入 warning，并使用基础 POI 信息 |
| Qdrant collection 不存在 | 自动创建集合 |
| Embedding 初始化失败 | 根据 `EMBED_PROVIDER` 尝试本地或 OpenAI-Compatible 后端 |
| 质量审核不通过 | 最多自动回流重跑 2 次，避免死循环 |
| 用户不继续精修 | 图停在 `interrupt`，已返回的行程仍可正常使用 |

## 13. 项目亮点总结

- 多 Agent 状态图编排与人机协同精修：基于 LangGraph `StateGraph` 设计旅行规划工作流，将系统拆分为知识检索、POI 筛选、实时信息采集、行程生成、图片增强、质量审核、用户精修等节点；结合 `MemorySaver + interrupt + Command` 实现 Human-in-the-loop 多轮精修，在生成行程后暂停并返回前端，用户提交自然语言修改意见后恢复同一会话状态继续执行。
- LLM Router 路由决策：设计独立的精修意图识别节点，调用大模型将用户反馈分类为消费档次调整、POI 增删替换、局部日程调整、全量重规划或结束流程，并动态路由至 `poi_selector`、`planner`、`knowledge_retrieval` 或 `END`。
- RAG 语义记忆：基于 LangChain + Qdrant 构建旅行攻略知识库，将本地文档切分、向量化并入库，在规划前检索相关城市攻略、避坑建议和偏好上下文，提升推荐结果的个性化与可信度。
- 用户情节记忆：设计 SQLite + Qdrant 双写机制，将用户评分、反馈文本和旅行城市结构化存储，并支持按 `user_id` 过滤召回历史经验，使后续规划能够参考用户过往偏好与负面反馈。
- 实时数据增强：接入高德地图 POI 和天气接口，使用异步 HTTP 批量查询景点、餐厅、酒店和天气信息，并对候选 POI 进行去重、限量筛选和兜底处理，提升行程数据真实性。
- 反思机制：设置 `qa_auditor` 独立质量审核节点，从地理合理性、预算匹配度、偏好满足度三个维度对行程评分，识别超预算、同日景点距离过远、偏好不匹配等问题；未通过时自动回流至 POI 筛选节点重跑。
- 地理优化：基于经纬度计算景点间距离，对每日景点进行就近排序，并为午餐、晚餐匹配当日景点附近餐厅，减少跨区域往返，提升行程可执行性。
- 前端可视化：开发 Vue3 + TypeScript 结果页，支持结构化行程展示、高德地图 `Marker/Polyline` 路线可视化、自然语言精修输入、行程反馈提交，以及 PNG/PDF 一键导出。
